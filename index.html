      🐳 docker · 01
      
Overview & Architecture

      
What Docker actually is, why teams use it, and the pieces that work together under the hood.

    


    

      
      

        
Simple way to think about it: A Virtual Machine is like building an entire separate house (own walls, own kitchen, own plumbing) just to host one guest. A container is like giving that guest their own furnished room in your existing house — much faster to set up, much lighter, and they share the building's foundation (the OS kernel).

        
    
          
  • VM: includes a full guest OS, takes minutes to boot, heavy (GBs), strong isolation via hypervisor.
    • 
          
  • Container: shares the host OS kernel, starts in seconds, lightweight (MBs), isolation via Linux namespaces & cgroups.
    • 
          
  • Why companies love it: "it works on my machine" problem disappears — the container ships the app + all its dependencies together, so it behaves the same on a laptop, test server, or production cloud.
    • 
          
  • Docker = a platform/tool that builds, ships and runs these containers using images.
    • 
        

      

    


    

      
      

        
Simple way to think about it: You (the CLI/client) place an order, the Docker Daemon is the kitchen that actually cooks (builds/runs) it, and the Registry (like Docker Hub) is the recipe book / warehouse where pre-made images are stored and pulled from.

        
    
          
  • Docker CLI: the docker command you type — sends instructions over a REST API.
    • 
          
  • Docker Daemon (dockerd): background service that does the real work — builds images, runs containers, manages networks & volumes.
    • 
          
  • containerd / runc: lower-level runtime that actually creates and runs the container processes.
    • 
          
  • Registry: stores images (Docker Hub, AWS ECR, GitHub Container Registry, private Harbor/Nexus).
    • 
          
  • Image: a read-only template/snapshot (your app + dependencies + OS layer).
    • 
          
  • Container: a running (or stopped) instance of an image — adds a thin writable layer on top.
    • 
        

        
# the typical flow in a CI/CD pipeline
docker build -t myorg/myapp:1.0 .
docker push myorg/myapp:1.0        # goes to the registry
docker pull myorg/myapp:1.0        # server pulls it down
docker run -d -p 8080:80 myorg/myapp:1.0

      

    


    

      
      

        
Simple way to think about it: An image is like a frozen, sealed lasagna recipe — every layer (base, sauce, cheese) is stacked and locked. A container is the actual hot lasagna on your plate, where you can still add extra cheese on top (the writable layer) without changing the original recipe.

        
    
          
  • Each instruction in a Dockerfile (FROM, RUN, COPY...) creates a new read-only layer.
    • 
          
  • Layers are cached — if a layer hasn't changed, Docker reuses it on the next build (this is why order of Dockerfile instructions matters for build speed).
    • 
          
  • A running container adds one thin writable layer on top — anything you write inside the container goes there, and is lost when the container is removed (unless using volumes).
    • 
          
  • docker commit can freeze a container's current state into a new image (rarely used in real workflows — prefer Dockerfiles for reproducibility).
    • 
        

      

    

  


  
  
  
  

    

      🐳 docker · 02
      
Images, Layers & Dockerfile

      
How to actually write a good Dockerfile, and the multi-stage trick that keeps images small in production.

    


    

      
      

        

          
InstructionWhat it does (in plain words)

          
FROMThe starting point/base image (e.g. node:20-alpine). Every Dockerfile starts here.

          
WORKDIRSets the "current folder" inside the image for following commands.

          
COPYCopies files from your machine into the image.

          
RUNExecutes a command during build (e.g. install packages). Creates a new layer.

          
ENVSets an environment variable that's available at build & run time.

          
ARGA build-time-only variable, not available once the container is running.

          
EXPOSEDocuments which port the app listens on (doesn't actually publish it — that's -p in docker run).

          
USERSwitches to a non-root user for security.

          
CMDThe default command to run when the container starts (can be overridden).

          
ENTRYPOINTThe fixed command that always runs; CMD args get appended to it.

        

        
CMD vs ENTRYPOINT (the classic confusion): Think of ENTRYPOINT as the verb ("run this script") that can't be changed easily, and CMD as the default arguments to that verb, which can be overridden when you run docker run myimage some-other-args.

      

    


    

      
      

        
Simple way to think about it: You don't ship the entire kitchen (compilers, build tools, source code) to the customer — you only ship the finished dish. Multi-stage build = "build in one container, copy only the final output into a clean, lightweight container."

        
# ---- Stage 1: build ----
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# ---- Stage 2: final, tiny image ----
FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

        
    
          
  • Result: final image only has nginx + built static files — not Node, npm, or your source code. Often shrinks images from 1GB+ down to under 50MB.
    • 
          
  • Use a small base image (alpine, slim, or distroless) wherever possible.
    • 
          
  • Always add a .dockerignore (just like .gitignore) so node_modules, .git, logs, etc. don't get copied in and bloat/leak into the image.
    • 
          
  • Order matters for caching: copy package.json and install dependencies before copying the rest of the source — so code changes don't force a full dependency reinstall.
    • 
          
  • Run as non-root: add RUN addgroup -S app && adduser -S app -G app then USER app for better security.
    • 
        

      

    


    

      
      

        
# build an image from Dockerfile in current dir
docker build -t myapp:1.0 .

# tag it for your registry (e.g. Docker Hub / ECR / GHCR)
docker tag myapp:1.0 myrepo/myapp:1.0

# push to registry — needs docker login first
docker push myrepo/myapp:1.0

# build without cache (force fresh rebuild — useful when debugging weird cache issues)
docker build --no-cache -t myapp:1.0 .

        
    
          
  • Tag convention in real teams: use :latest only for local testing — production deployments use immutable tags like :1.4.2 or :git-sha-abc123, so rollbacks are predictable.
    • 
        

      

    

  


  
  
  
  

    

      🐳 docker · 03
      
Networking & Volumes

      
How containers talk to each other, the outside world, and how data survives container restarts.

    


    

      
      

        
Simple way to think about it: By default every container gets its own private "apartment" with its own IP (bridge network). If two containers are on the same custom network, they can call each other by container name — Docker has a built-in DNS for that, like apartments on the same floor knowing each other's room numbers.

        

          
Network typeWhat it means

          
bridge (default)Containers get an internal private IP, isolated from host network. Default for standalone containers.

          
custom bridgeYou create it (docker network create) — containers on it can resolve each other by name. This is what you use in real projects.

          
hostContainer shares the host's network directly — no isolation, fastest, useful for performance-sensitive tools.

          
noneNo networking at all — fully isolated.

          
overlaySpans multiple Docker hosts (Swarm) — containers on different machines talk as if on the same network.

        

        
# create a custom network so containers can find each other by name
docker network create app-net

docker run -d --name db --network app-net postgres:16
docker run -d --name api --network app-net myapp:1.0
# inside "api" container, connect to DB using hostname "db" — no IP needed!

      

    


    

      
      

        
Simple way to think about it: A container's own filesystem is like a whiteboard that gets wiped clean every time the container is removed. A volume is an external hard drive Docker manages for you and plugs into the container. A bind mount is plugging in a specific folder from your laptop/server directly.

        

          
TypeManaged byCommon use

          
Named volumeDocker (in /var/lib/docker/volumes)Databases, persistent app data — portable, backed-up easily.

          
Bind mountYou (any host path)Local development — live-reload source code into the container.

          
tmpfsRAM only, never written to diskTemporary secrets/cache that should never persist.

        

        
# named volume — survives container removal
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres:16

# bind mount — great for local dev, live code reload
docker run -d -v $(pwd)/src:/app/src myapp:1.0

# list / inspect / remove volumes
docker volume ls
docker volume inspect pgdata
docker volume rm pgdata

      

    


    

      
      

        
Simple way to think about it: EXPOSE in a Dockerfile is just a label/note saying "this app talks on port 80" — like a sign on a door. -p at runtime is the actual action of cutting a hole in the wall so the outside world can reach that door.

        
# HOST_PORT : CONTAINER_PORT
docker run -d -p 8080:80 nginx
# now http://localhost:8080 -> container's port 80

# bind only to localhost (safer for internal services)
docker run -d -p 127.0.0.1:8080:80 nginx

      

    

  


  
  
  
  

    

      🐳 docker · 04
      
Docker Compose

      
Running a whole multi-container app (app + db + cache) with one command — the everyday dev workflow.

    


    

      
      

        
Simple way to think about it: Compose is a shopping list + setup instructions for your whole app stack. Instead of running 4 separate docker run commands with long flags, you describe everything once in a YAML file and run docker compose up.

        
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=db
      - DB_PASSWORD=${DB_PASSWORD}
    depends_on:
      - db
    volumes:
      - ./src:/app/src   # live reload in dev

  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine

volumes:
  pgdata:

        
    
          
  • depends_on controls start order only — it does NOT wait for the app inside to actually be "ready" (that's what healthchecks + readiness logic is for).
    • 
          
  • ${DB_PASSWORD} pulls values from a .env file sitting next to the compose file — never hardcode secrets in the YAML.
    • 
          
  • All services on the same compose file are automatically put on one shared network and can reach each other by service name (db, redis).
    • 
        

      

    


    

      
      

        
docker compose up -d            # start everything in background
docker compose up -d --build    # rebuild images before starting
docker compose ps               # list running services
docker compose logs -f api      # follow logs for one service
docker compose exec api sh      # open a shell inside running service
docker compose down             # stop & remove containers + network
docker compose down -v          # also delete volumes (careful — deletes DB data!)
docker compose restart api      # restart just one service

Command Cheat Sheet

      
The commands you'll type without thinking, every single day.

docker run -d --name api -p 8080:80 myappStart a container in background, named "api", with port mapping.

          
docker psList running containers.

          
docker ps -aList ALL containers, including stopped ones.

          
docker logs -f apiStream logs live (-f = follow).

          
docker exec -it api shOpen an interactive shell inside a running container.

          
docker stop apiGracefully stop a container (SIGTERM, then SIGKILL after timeout).

          
docker start apiStart a previously stopped container.

          
docker restart apiStop + start again.

          
docker rm apiRemove a stopped container.

          
docker rm -f apiForce-remove even if running.

          
docker inspect apiFull JSON details — IP, mounts, env vars, etc.

          
docker statsLive CPU/memory usage per container.

        

      

    


    

      
      

        

          
CommandWhat it does

          
docker image lsList all images and their sizes.

      🐳 docker · 06 · ⚠ daily ops
      
Real Office Issues — Docker

      
The exact problems that show up in Slack/Teams on a normal Tuesday, with how to find and fix them.

    


    

      
IssueContainer exits immediately after docker run (Exited (1) / Exited (0))▾

      

        
Why it happens

        
The main process inside the container finished or crashed. Containers stay alive only as long as their PID 1 process is running — there's no "background OS" keeping it alive like a VM.

        
How to investigate

        
docker ps -a                 # find the container, note its exit code
docker logs <container>      # 90% of the time the error is right here
docker run -it myimage sh    # run interactively to poke around

        
Common root causes

        
    
          
  • App crashed on startup (missing env var, bad config, DB not reachable yet).
    • 
          
  • The Dockerfile's CMD was something like CMD ["bash"] with no -it — bash exits instantly with nothing to do.
    • 
          
  • Exit code 0 = process finished normally (maybe it was a one-off script, not a server).
    • 
          
  • Exit code 137 = killed by OOM (out of memory) — check docker inspect for OOMKilled: true.
    • 
        

      

    


    

      
Issue"Bind for 0.0.0.0:8080 failed: port is already allocated"▾

      

        
Why it happens

        
Something — another container, or a process on the host — is already using that port.

        
How to fix

        
docker ps                       # check if another container already uses 8080
sudo lsof -i :8080              # find the host process using the port (Linux/Mac)
docker stop <old_container>     # stop the conflicting container
# OR simply map to a different host port:
docker run -d -p 8081:80 myapp

      

    


    

      
IssueImage is huge (1GB+) and slow to push/pull/deploy▾

      

        
Why it happens

        
Using a heavy base image (full Ubuntu/Debian instead of alpine), copying unnecessary files, leaving build tools/dev dependencies in the final image, or not cleaning up package manager caches.

        
How to fix

        
    
          
  • Switch base image to -alpine or -slim variants.
    • 
          
  • Use a multi-stage build (see Images & Dockerfile section) — final stage only contains runtime essentials.
    • 
          
  • Add a proper .dockerignore (exclude node_modules, .git, build artifacts, logs).
    • 
          
  • Combine RUN commands and clean up in the same layer: RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*
    • 
          
  • Run docker history myimage to see which layer is the biggest culprit.
    • 
        

      

    


    

      
IssueApp container can't connect to the database container ("connection refused" / "host not found")▾

      

        
Why it happens

        
Using localhost inside a container refers to the container itself, not the host or other containers. Also: containers must be on the same Docker network to resolve each other by name, and the DB might just not be ready yet.

        
How to fix

        
# wrong — "localhost" inside the api container is the api container itself
DB_HOST=localhost

# correct — use the OTHER container's service/container name
DB_HOST=db

# verify both containers are on the same network
docker network inspect app-net

# test connectivity from inside the container
docker exec -it api sh -c "nc -zv db 5432"

        
    
          
  • For the "DB not ready yet" timing issue, add retry logic in the app or use a wait-for-it script / healthcheck-based startup ordering.
    • 
        

      

    


    

      
Issue"Permission denied" when writing to a mounted volume▾

      

        
Why it happens

        
The user inside the container (often a non-root user you added for security) doesn't have permission to write to the host folder — UID/GID mismatch between container user and host folder owner.

        
How to fix

        
    
          
  • Match UID/GID: run container with --user $(id -u):$(id -g) during local dev.
    • 
          
  • Or chown the host folder to match the container's user ID.
    • 
          
  • Or in the Dockerfile, create the app user with a specific UID: RUN adduser -u 1000 -D appuser then USER appuser.
    • 
        

      

    


    

      
Issue"No space left on device" on the build/CI server▾

      

        
Why it happens

        
Old images, stopped containers, build caches and unused volumes pile up over time — especially on CI runners that build images all day.

        
How to fix

        
docker system df               # see what's eating space
docker system prune -af        # remove unused images/containers/networks
docker volume prune -f         # remove unused volumes (check first!)
docker builder prune -af        # clear build cache

        
In CI pipelines, schedule this as a periodic cleanup job on the runners.

      

    


    

      
IssueEnvironment variables aren't being picked up by the app▾

      

        
Why it happens

        
Mixing up ARG vs ENV (ARG doesn't exist at runtime), wrong .env file path in Compose, or the app reading variables before they're set.

        
How to fix

        
    
          
  • ARG values only exist during docker build — if the app needs the value at runtime, also set it via ENV.
    • 
          
  • Confirm what the container actually sees: docker exec api env
    • 
          
  • In Compose, .env must sit in the same folder as docker-compose.yml (or pass --env-file explicitly).
    • 
          
  • Double-check there's no typo / case-sensitivity mismatch between the variable name in code vs the env file.
    • 
        

      

    


    

      
IssueCode changes aren't showing up after redeploying the container▾

      

        
Why it happens

        
Images are immutable snapshots — if you change source code but don't rebuild the image, the old code is still inside the running container. Also Docker's build cache can serve a stale COPY layer.

        
How to fix

        
docker compose up -d --build         # force rebuild before restarting
docker build --no-cache -t myapp .   # bypass cache entirely if a layer seems stuck

        
For local dev, use a bind mount of the source folder so changes appear instantly without rebuilding.

      

    

  


  
  
  
  

    

      🐳 docker · 07 · ❓ interview prep
      
Interview Q&A — Docker

      
Questions that come up again and again, answered the way you'd actually say them out loud.

    


    

      
Q.What's the difference between CMD and ENTRYPOINT?▾

      

        
A: ENTRYPOINT defines the fixed command that always runs; CMD provides default arguments that can be overridden at docker run time. If both are set, the CMD (or anything passed after the image name in docker run) is appended as arguments to ENTRYPOINT. Example: ENTRYPOINT ["python", "app.py"] with CMD ["--port=8080"] — running docker run myimage --port=9090 overrides just the port.

      

    


    

      
Q.What's the difference between COPY and ADD?▾

      

        
A: Both copy files from build context into the image. ADD additionally supports auto-extracting tar archives and fetching from URLs. Best practice: always prefer COPY for clarity/predictability, and only use ADD when you specifically need its extra extraction behavior.

      

    


    

      
Q.What is the difference between an image and a container?▾

      

        
A: An image is a read-only, immutable template/blueprint (built from a Dockerfile). A container is a running (or stopped) instance of that image, with an added writable layer. You can run many containers from the same image — same analogy as a class (image) vs an object/instance (container).

      

    


    

      
Q.How would you reduce the size of a Docker image?▾

      

        
A: Use a smaller base image (alpine/slim/distroless), use multi-stage builds so build tools never reach the final image, minimize layers by chaining RUN commands, clean package manager caches in the same layer they're created, add a .dockerignore, and avoid copying unnecessary files like .git or test fixtures.

      

    


    

      
Q.Explain bridge, host, and overlay networks.▾

      

        
A: bridge is the default isolated network on a single host — containers get private IPs and a custom bridge network adds DNS-based name resolution between containers. host removes network isolation entirely — the container shares the host's network stack directly (max performance, no port mapping needed). overlay connects containers across multiple Docker hosts (used in Swarm/multi-host setups), letting them communicate as if on one network.

      

    


    

      
Q.Volume vs bind mount — when would you use each?▾

      

        
A: Named volumes are managed by Docker, portable, and the right choice for persistent data like databases in production. Bind mounts map a specific host path into the container and are mainly used in local development for live code reload, or to give a container access to specific host files (like a config file or Docker socket).

      

    


    

      
Q.What is a multi-stage build and why use it?▾

      

        
A: It's a Dockerfile with multiple FROM stages — earlier stages can compile/build the app with all the heavy tooling, and the final stage copies only the build output (COPY --from=build ...) into a clean, minimal runtime image. This drastically reduces image size and removes build tools/source code from production images, improving both performance and security.

      

    


    

      
Q.Docker Compose vs Docker Swarm vs Kubernetes — how do they compare?▾

      

        
A: Compose runs a multi-container app on a single host — great for local dev/small setups. Swarm and Kubernetes are orchestrators that manage containers across multiple hosts/nodes — handling scheduling, scaling, self-healing, rolling updates, and service discovery. Kubernetes is the industry-standard, far more feature-rich and widely adopted than Swarm, which has largely fallen out of favor.

      

    


    

      
Q.How does Docker's build cache work, and how do you take advantage of it?▾

      

        
A: Each Dockerfile instruction produces a layer; Docker caches each layer and reuses it if the instruction and its input (files for COPY, command for RUN) haven't changed since the last build. Once one layer changes, every layer after it must be rebuilt. So you order instructions from least-to-most frequently changing — e.g. install dependencies before copying application source code, so a source code change doesn't force a full dependency reinstall.

      

    


    

      
Q.How would you debug a container that keeps crashing?▾

      

        
A: Start with docker ps -a to see the exit code, then docker logs <container> for the error message — this solves most issues. If the container exits too fast to inspect, override the entrypoint and run it interactively: docker run -it --entrypoint sh myimage to poke around the filesystem and try running the command manually. Check docker inspect for OOMKilled, exit code, and mounted volumes/env vars for misconfiguration.

      

    

  



  
  
  
  

    

      ☸️ kubernetes · 01
      
Cluster Architecture

      
The "company org chart" of a Kubernetes cluster — who does what, and why it matters when something breaks.

    


    

      
      

        
Simple way to think about it: Think of a cluster like a delivery company. The Control Plane is head office — it takes orders, decides which driver delivers what, and keeps records. Worker Nodes are the delivery vans actually carrying and delivering the packages (your containers).

        

          
ComponentLives onPlain-English job

          
API ServerControl planeThe front door — every kubectl command and every internal component talks to the cluster through here.

          
etcdControl planeThe cluster's database — stores the entire desired state (what should be running, configs, secrets).

          
SchedulerControl planeDecides which node a new Pod should run on, based on resources, constraints, etc.

          
Controller ManagerControl planeRuns background loops that constantly check "is reality matching the desired state?" and fixes drift (e.g. restarts a crashed Pod).

          
kubeletEvery worker nodeThe agent on each node that actually starts/stops containers as instructed, and reports node health back.

          
kube-proxyEvery worker nodeHandles network rules so Services can route traffic to the right Pods.

          
Container RuntimeEvery worker nodeThe software that actually runs containers (containerd, CRI-O).

        

      

    


    

      
      

        
Simple way to think about it: You tell Kubernetes "I always want 3 copies of this app running" (the desired state) — and it constantly checks reality against that. If a Pod dies, it's like a manager who notices "we're supposed to have 3 staff on shift but only 2 showed up" and immediately calls in a replacement — automatically, without anyone asking.

        
    
          
  • You write YAML manifests describing what you want (declarative), not step-by-step commands (imperative).
    • 
          
  • kubectl apply -f deployment.yaml sends this desired state to the API server, which stores it in etcd.
    • 
          
  • Controllers continuously "reconcile" — comparing actual cluster state vs desired state, and taking action to close any gap.
    • 
          
  • This is why Kubernetes can self-heal: kill a Pod manually, and a Deployment controller notices the replica count dropped and creates a new one within seconds.
    • 
        

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 02
      
Pods, Deployments & Workloads

      
The building blocks you'll define in nearly every YAML file you write.

    


    

      
      

        
Simple way to think about it: A Pod is like a small shared apartment. Usually one tenant (container) lives there, but sometimes a roommate (sidecar container) moves in too — they share the same address (network/IP) and some common areas (storage volumes).

        
    
          
  • A Pod wraps one or more containers that are always scheduled together on the same node, sharing network namespace (same IP/localhost) and optionally storage.
    • 
          
  • Sidecar pattern: e.g. main app container + a logging agent container in the same Pod — common for log shipping, service mesh proxies (Istio), config reloaders.
    • 
          
  • Pods are ephemeral — you almost never create raw Pods directly in production; instead you use a Deployment/StatefulSet/Job that manages Pods for you.
    • 
        

        
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app: myapp
spec:
  containers:
    - name: myapp
      image: myorg/myapp:1.0
      ports:
        - containerPort: 8080

      

    


    

      
      

        
Simple way to think about it: A ReplicaSet is a manager whose only job is "always keep exactly N copies of this Pod running." A Deployment is the manager's manager — it controls ReplicaSets and handles the smooth process of rolling out a new version (creating a new ReplicaSet, scaling it up while scaling the old one down).

        
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myorg/myapp:1.4.2
          ports:
            - containerPort: 8080

        

          
CommandWhat it does

          
kubectl rollout status deployment/myappWatch a rollout in progress.

          
kubectl rollout history deployment/myappSee previous revisions.

          
kubectl rollout undo deployment/myappRoll back to the previous version — your "undo button" for bad deploys.

          
kubectl set image deployment/myapp myapp=myorg/myapp:1.4.3Update the image and trigger a rolling update.

        

        
    
          
  • RollingUpdate (default): replaces Pods gradually — zero downtime. maxUnavailable / maxSurge control how aggressive the rollout is.
    • 
          
  • Recreate: kills all old Pods first, then creates new ones — causes downtime, used when the app can't run two versions side-by-side.
    •

DaemonSetRuns exactly one Pod on every node automatically.Log collectors, monitoring agents (e.g. Fluentd, Node Exporter), network plugins.

  jobTemplate:
    spec:

Simple way to think about it: Labels are like sticky notes on boxes ("env=prod", "app=myapp", "tier=frontend"). Selectors are the instruction "find me all boxes with the sticky note 'app=myapp'" — this is exactly how a Service finds the right Pods, or a Deployment knows which Pods belong to it.

        
    
          
  • Labels: key/value pairs attached to objects, used for grouping & selecting (e.g. app: myapp, env: prod, tier: backend).
    • 
          
  • Selectors: queries that match labels — Services, Deployments, and NetworkPolicies all use selectors to find their target Pods.
    • 
          
  • Annotations: also key/value, but NOT used for selection — used for metadata like descriptions, build info, or config read by tools (e.g. ingress controller annotations).
    • 
        

        
kubectl get pods -l app=myapp,env=prod   # filter by labels
kubectl label pod myapp-pod tier=frontend   # add a label

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 03
      
Services & Ingress

      
How traffic actually finds its way to your Pods — internally and from the outside world.

    


    

      
      

        
Simple way to think about it: Pods are like employees who get a new desk (IP address) every time they come back from a break — unreliable to call directly. A Service is a permanent reception desk with one fixed phone number that always knows which employees (Pods) are currently "at work" and forwards your call to one of them.

        

          
TypeWhat it doesUsed for

          
ClusterIP (default)Internal-only stable IP, reachable only inside the cluster.Service-to-service communication (e.g. API → DB).

          
NodePortOpens a static port (30000-32767) on every node, forwarding to the Service.Quick external access / on-prem setups without a cloud load balancer.

          
LoadBalancerCloud provider provisions an external load balancer pointing at the Service.Production external access on AWS/GCP/Azure.

          
Headless (clusterIP: None)No single virtual IP — DNS returns the IPs of all individual Pods directly.StatefulSets where you need to address each Pod individually (e.g. each DB replica).

        

        
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  selector:
    app: myapp        # finds Pods with label app=myapp
  ports:
    - port: 80           # the Service's own port
      targetPort: 8080    # the container's port
  type: ClusterIP

        
Other Pods reach this Service via DNS: http://myapp-svc (same namespace) or http://myapp-svc.namespace.svc.cluster.local (cross-namespace).

      

    


    

      
      

        
Simple way to think about it: If each Service is a department's internal extension number, an Ingress is the building's main receptionist — one single entry point (one IP/Load Balancer) that reads the visitor's request ("I'm here for api.example.com/orders") and routes them to the right department.

        
    
          
  • Without Ingress, exposing 10 services externally would mean 10 separate LoadBalancers (expensive). Ingress lets one entry point route by hostname/path to many Services.
    • 
          
  • An Ingress Controller (NGINX, Traefik, AWS ALB Controller, etc.) must be installed in the cluster — the Ingress resource itself is just rules; the controller does the actual routing.
    • 
          
  • Also commonly handles TLS/SSL termination (HTTPS certificates).
    • 
        

        
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /orders
            pathType: Prefix
            backend:
              service:
                name: orders-svc
                port:
                  number: 80
          - path: /users
            pathType: Prefix
            backend:
              service:
                name: users-svc
                port:
                  number: 80
  tls:
    - hosts: ["api.example.com"]
      secretName: api-tls-cert

      

    


    

      
      

        
Simple way to think about it: Every Service automatically gets a "company directory" entry. Any Pod can call another service by name (like calling a coworker by their name instead of memorizing their desk phone extension), and CoreDNS (the cluster's internal phonebook) resolves that name to the right IP.

        
    
          
  • CoreDNS runs as cluster add-on and resolves Service names to ClusterIPs automatically.
    • 
          
  • Full DNS format: <service-name>.<namespace>.svc.cluster.local — within the same namespace, just <service-name> works.
    • 
          
  • Environment variables for Services are also injected automatically into Pods (legacy mechanism, DNS is preferred).
    • 
        

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 04
      
Config, Secrets & Storage

      
Keeping configuration, sensitive data, and persistent data separate from your application images.

    


    

      
      

        
Simple way to think about it: A ConfigMap is a sticky note with non-sensitive settings (like "THEME=dark", "API_TIMEOUT=30s") pinned to the fridge for anyone to read. A Secret is the same idea but for things like passwords — stored in a locked drawer (base64-encoded by default, and ideally encrypted at rest + access-controlled via RBAC).

        
# ConfigMap — non-sensitive config
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_THEME: "dark"
  API_TIMEOUT: "30"

---
# Secret — sensitive data (base64 encoded, NOT encrypted by default)
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  DB_PASSWORD: cGFzc3dvcmQxMjM=   # base64 of "password123"

        
# using them inside a Pod
spec:
  containers:
    - name: myapp
      image: myapp:1.0
      envFrom:
        - configMapRef: { name: app-config }
        - secretRef: { name: db-secret }
      # OR mount as files in a volume — useful for cert files/config files
      volumeMounts:
        - name: config-vol
          mountPath: /etc/config
  volumes:
    - name: config-vol
      configMap: { name: app-config }

        
    
          
  • Important: Secrets are base64-encoded, not encrypted — anyone with read access to the Secret object can decode it. Real protection comes from RBAC + enabling encryption-at-rest in etcd, or using tools like Sealed Secrets / Vault.
    • 
          
  • Updating a ConfigMap/Secret doesn't automatically restart Pods using it as env vars — see the Real Office Issues section for the fix.
    • 
        

      

    


    

      
      

        
Simple way to think about it: A PersistentVolume (PV) is an actual storage unit in a self-storage facility. A PersistentVolumeClaim (PVC) is the rental agreement your app fills out saying "I need 10GB of storage" — Kubernetes then matches it to an available unit (or creates one on demand via a StorageClass, like an automated booking system).

        
# PVC — your app's storage "order"
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: db-pvc
spec:
  accessModes: ["ReadWriteOnce"]
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi

---
# mounted into a Pod / StatefulSet
spec:
  containers:
    - name: postgres
      image: postgres:16
      volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: db-pvc

        
    
          
  • ReadWriteOnce (RWO): mountable by one node at a time (most cloud block storage). ReadWriteMany (RWX): multiple nodes can mount simultaneously (needs NFS/EFS-style storage).
    • 
          
  • StorageClass defines how storage gets dynamically provisioned (e.g. AWS gp3 EBS, GCP pd-ssd) — without it, an admin would have to pre-create PVs manually.
    • 
          
  • Data in a PV survives Pod restarts/rescheduling — unlike a Pod's own filesystem.
    • 
        

      

    


    

      
      

        
Simple way to think about it: A cluster is one big office building. Namespaces are like separate floors for different teams (dev, staging, prod) — names only need to be unique within a floor, and you can set rules like "this floor can't use more than 10 desks total" (Resource Quotas).

        
kubectl create namespace staging
kubectl get pods -n staging
kubectl config set-context --current --namespace=staging   # switch default namespace

        
    
          
  • Built-in namespaces: default, kube-system (cluster components), kube-public.
    • 
          
  • Some resources are namespaced (Pods, Services, Deployments, ConfigMaps) and some are cluster-scoped (Nodes, PersistentVolumes, Namespaces themselves, ClusterRoles).
    • 
          
  • ResourceQuota objects limit total CPU/memory/object counts per namespace — common in shared clusters to prevent one team hogging resources.
    • 
        

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 05
      
Health, Resources & Scaling

      
How Kubernetes knows if your app is healthy, how much it can use, and when to add more copies.

    


    

      
      

        
Simple way to think about it: Imagine a waiter in a restaurant. Liveness = "is this waiter still conscious? If not, replace them" (restart the container). Readiness = "is this waiter currently free to take an order, or busy/on a break?" (temporarily remove from the Service's pool, but don't restart). Startup = "give this trainee waiter extra time to get ready before we start checking on them at all."

        

          
ProbeIf it fails...Use for

          
livenessProbeContainer is restartedDetecting a hung/deadlocked process that needs a restart.

          
readinessProbePod is removed from Service endpoints (no restart)App temporarily can't serve traffic (e.g. still loading cache, DB connection lost).

          
startupProbeDisables liveness/readiness checks until it passesSlow-starting apps (legacy Java apps) that would otherwise get killed by liveness too early.

        

        
spec:
  containers:
    - name: myapp
      image: myapp:1.0
      livenessProbe:
        httpGet: { path: /healthz, port: 8080 }
        initialDelaySeconds: 10
        periodSeconds: 10
        failureThreshold: 3
      readinessProbe:
        httpGet: { path: /ready, port: 8080 }
        periodSeconds: 5
      startupProbe:
        httpGet: { path: /healthz, port: 8080 }
        failureThreshold: 30
        periodSeconds: 10

      

    


    

      
      

        
Simple way to think about it: requests = the minimum table size you reserve at a restaurant (guarantees you get seated). limits = the maximum you're allowed to eat — if you go over on memory, you get kicked out (OOMKilled); if you go over on CPU, you just get slowed down (throttled), not removed.

        
resources:
  requests:
    cpu: "250m"     # 0.25 of a CPU core — used for scheduling decisions
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"  # exceed this -> OOMKilled

        
    
          
  • The Scheduler uses requests to decide which node has enough free capacity for a Pod.
    • 
          
  • Exceeding memory limit → container is killed with OOMKilled. Exceeding CPU limit → container is throttled, not killed.
    • 
          
  • QoS classes are derived automatically: Guaranteed (requests == limits for all resources), Burstable (requests < limits), BestEffort (no requests/limits set — first to be killed under node pressure).
    • 
        

      

    


    

      
      

        
Simple way to think about it: HPA is like a call-center manager watching the queue — if calls (CPU/traffic) pile up beyond a threshold, more agents (Pods) are brought on; when it's quiet, agents are sent home, within set min/max limits.

        
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

        
    
          
  • Requires the metrics-server add-on to be installed to read CPU/memory usage.
    • 
          
  • VPA (Vertical Pod Autoscaler) adjusts a Pod's resource requests/limits over time instead of adding more Pods — less commonly used together with HPA on the same metric.
    • 
          
  • kubectl scale deployment myapp --replicas=5 — manual scaling, useful for quick fixes before automation kicks in.
    • 
        

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 06
      
RBAC & Helm

      
Controlling who can do what, and packaging up your YAML so it's reusable.

    


    

      
      

        
Simple way to think about it: A Role is a job description ("can read and list Pods in this namespace"). A RoleBinding is HR assigning that job description to a specific person or service account. A ServiceAccount is an identity badge that a Pod wears so it can authenticate itself to the API server.

        
# Role: defines WHAT can be done (namespace-scoped)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]

---
# RoleBinding: defines WHO gets that Role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: dev
subjects:
  - kind: ServiceAccount
    name: ci-bot
    namespace: dev
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

        
    
          
  • Role = namespace-scoped permissions. ClusterRole = cluster-wide (or reusable across namespaces) permissions.
    • 
          
  • RoleBinding = grants within one namespace. ClusterRoleBinding = grants cluster-wide.
    • 
          
  • Every Pod runs with a ServiceAccount (default: default) — give CI/CD pipelines and apps their own dedicated ServiceAccounts with least-privilege Roles.
    • 
        

      

    


    

      
      

        
Simple way to think about it: Writing raw YAML for every environment (dev/staging/prod) means copy-pasting and tweaking dozens of files. Helm is like a recipe template with fill-in-the-blank ingredients (values.yaml) — one chart, different values per environment.

        
helm install myapp ./mychart -f values-prod.yaml
helm upgrade myapp ./mychart -f values-prod.yaml
helm rollback myapp 1          # roll back to revision 1
helm list                       # see installed releases
helm uninstall myapp

        
    
          
  • A Chart = a packaged set of templated YAML manifests + default values.yaml.
    • 
          
  • {{ .Values.replicaCount }} style placeholders in templates get filled in from values.yaml (or overridden with -f / --set).
    • 
          
  • Each helm install/upgrade creates a release — tracked with its own revision history, enabling easy rollbacks.
    •

      ☸️ kubernetes · 08 · ⚠ daily ops
      
Real Office Issues — Kubernetes

      
The pager-duty classics. Every one of these will happen to you for real — here's the playbook.

    


    

      
IssueCrashLoopBackOff▾

      

        
What it means

        
The container starts, crashes, Kubernetes restarts it, it crashes again — and Kubernetes increases the wait time between restarts each time (backoff).

        
How to investigate

        
kubectl get pods                              # confirm the status
kubectl logs <pod>                            # current attempt's logs
kubectl logs <pod> --previous                 # logs from the crashed attempt — usually the real error
kubectl describe pod <pod>                    # check Events section at the bottom

        
Common causes

        
    
          
  • Application error on startup (bad config, missing env var, can't connect to DB).
    • 
          
  • Liveness probe failing repeatedly and killing the container before it can finish starting (fix with a startupProbe or longer initialDelaySeconds).
    • 
          
  • Wrong command/entrypoint in the image causing it to exit immediately.
    • 
        

      

    


    

      
IssueImagePullBackOff / ErrImagePull▾

      

        
What it means

        
The node can't pull the container image specified in the Pod spec.

        
How to investigate

        
kubectl describe pod <pod>   # Events will show the exact pull error

        
Common causes

        
    
          
  • Typo in image name/tag — e.g. myrepo/myapp:1.0.0 vs the actual 1.0.
    • 
          
  • Private registry without credentials — need an imagePullSecrets referencing a Docker registry Secret on the Pod spec.
    • 
          
  • Rate limiting from Docker Hub on anonymous pulls.
    • 
          
  • Image was deleted/never pushed to the registry.
    • 
        

        
spec:
  imagePullSecrets:
    - name: regcred
  containers:
    - name: myapp
      image: myprivateregistry.com/myapp:1.0

      

    


    

      
IssuePod stuck in "Pending"▾

      

        
What it means

        
The Pod has been accepted but the Scheduler hasn't been able to place it on any node yet.

        
How to investigate

        
kubectl describe pod <pod>   # "Events" section explains exactly why scheduling failed
kubectl get nodes            # check node status & available capacity
kubectl describe nodes | grep -A5 Allocated

        
Common causes

        
    
          
  • Insufficient CPU/memory on all nodes to satisfy the Pod's requests.
    • 
          
  • Node selectors / affinity rules / taints — Pod requires a node type that doesn't exist or isn't tolerated.
    • 
          
  • PVC not bound — Pod is waiting for storage to be provisioned.
    • 
          
  • Hitting a ResourceQuota limit in the namespace.
    • 
        

      

    


    

      
IssueOOMKilled▾

      

        
What it means

        
The container tried to use more memory than its configured limit (or the node ran out of memory), so the kernel killed it.

        
How to investigate

        
kubectl describe pod <pod>   # Look for "Last State: Terminated, Reason: OOMKilled"
kubectl top pod <pod>        # check current memory usage vs limit

        
How to fix

        
    
          
  • Increase the memory limit if the app genuinely needs more.
    • 
          
  • Investigate a memory leak in the app if usage keeps climbing over time.
    • 
          
  • Check if requests were set too low, causing the scheduler to overcrowd a node, leading to node-level memory pressure evicting Pods.
    • 
        

      

    


    

      
IssueService exists, Pods are Running, but the app isn't reachable▾

      

        
Why it happens

        
This is almost always a label/selector mismatch, a wrong targetPort, or a failing readiness probe keeping Pods out of the Service's endpoint list.

        
How to investigate

        
# Step 1: does the Service have any endpoints at all?
kubectl get endpoints myapp-svc
# If empty -> Service selector doesn't match any Pod labels

# Step 2: compare selector vs pod labels
kubectl get svc myapp-svc -o yaml | grep -A3 selector
kubectl get pods --show-labels

# Step 3: check targetPort matches the container's actual listening port
kubectl describe pod <pod> | grep -i port

# Step 4: test from inside the cluster
kubectl run debug --rm -it --image=busybox -- sh
# then inside: wget -O- http://myapp-svc

        
Common causes

        
    
          
  • Service selector labels don't exactly match the Pod's labels (case-sensitive, exact match).
    • 
          
  • targetPort doesn't match the port the app actually listens on inside the container.
    • 
          
  • Readiness probe is failing, so the Pod is Running but never added to the Service's Endpoints.
    • 
        

      

    


    

      
IssueUpdated a ConfigMap/Secret, but the app is still using old values▾

      

        
Why it happens

        
If the ConfigMap/Secret is consumed as environment variables, those are injected only once at Pod creation — updating the ConfigMap does NOT update env vars in already-running Pods.

        
How to fix

        
# trigger a rolling restart so Pods pick up new env values
kubectl rollout restart deployment myapp

        
    
          
  • If mounted as a volume (file), kubelet eventually syncs the updated file into the Pod automatically (with some delay) — but the app itself usually still needs to reload the file (most apps don't watch config files for changes).
    • 
          
  • For zero-downtime config reloads, consider tools like Reloader which auto-restart Pods when their ConfigMap/Secret changes, or build a sidecar that signals the app to reload.
    • 
        

      

    


    

      
IssueNode shows "NotReady"▾

      

        
Why it happens

        
The kubelet on that node has stopped reporting heartbeats — could be network issues, kubelet crashed, disk pressure, or the node itself is down.

        
How to investigate

        
kubectl get nodes
kubectl describe node <node>   # check Conditions: MemoryPressure, DiskPressure, etc.

        
What happens to Pods

        
    
          
  • After a tolerance period (default 5 minutes), Pods on a NotReady node are evicted and rescheduled onto healthy nodes (if managed by a Deployment/ReplicaSet).
    • 
          
  • To safely take a node down for maintenance: kubectl cordon <node> (stop new Pods being scheduled there) then kubectl drain <node> (evict existing Pods gracefully).
    • 
        

      

    


    

      
IssuePVC stuck in "Pending"▾

      

        
Why it happens

        
No PersistentVolume could be created/matched to satisfy the claim.

        
How to investigate

        
kubectl describe pvc <pvc-name>   # Events will show the reason
kubectl get storageclass            # confirm the requested StorageClass exists

        
Common causes

        
    
          
  • Typo'd or missing storageClassName.
    • 
          
  • Requested accessModes (e.g. ReadWriteMany) isn't supported by the underlying storage provider.
    • 
          
  • Cloud quota limits reached (e.g. max EBS volumes for the account/region).
    • 
        

      

    


    

      
IssueDNS resolution failing inside a Pod (can't resolve other Services)▾

      

        
How to investigate

        
kubectl exec -it <pod> -- nslookup myapp-svc
kubectl get pods -n kube-system -l k8s-app=kube-dns   # is CoreDNS healthy?
kubectl logs -n kube-system -l k8s-app=kube-dns

        
Common causes

        
    
          
  • CoreDNS Pods are down/CrashLooping.
    • 
          
  • Using the wrong name format — remember cross-namespace lookups need service.namespace or the full service.namespace.svc.cluster.local.
    • 
          
  • NetworkPolicy accidentally blocking traffic to CoreDNS (port 53).
    • 
        

      

    


    

      
IssueA perfectly healthy app keeps getting restarted by its liveness probe▾

      

        
Why it happens

        
The probe's timing doesn't match the app's real behavior — e.g. the app takes 20s to fully start, but initialDelaySeconds is only 5s, so the liveness probe fails and restarts it before it's even up — an endless loop.

        
How to fix

        
    
          
  • Add/increase a startupProbe with a generous failureThreshold × periodSeconds for slow-starting apps — liveness/readiness won't run until startup succeeds.
    • 
          
  • Increase timeoutSeconds if the health endpoint is sometimes slow under load (e.g. it does a DB ping that occasionally takes >1s).
    • 
          
  • Make sure the health-check endpoint itself is lightweight and doesn't depend on things that can be temporarily down (don't fail liveness just because a downstream cache is briefly unreachable — that's a job for readiness, not liveness).
    • 
        

      

    


    

      
IssueDeployment rollout stuck — new Pods never become Ready▾

      

        
How to investigate

        
kubectl rollout status deployment/myapp
kubectl get pods -l app=myapp           # check status of new ReplicaSet's pods
kubectl describe pod <new-pod>          # readiness probe failing? crashing?

        
Common causes

        
    
          
  • New image version has a bug (CrashLoopBackOff on new Pods only) — fix: kubectl rollout undo deployment/myapp to roll back immediately while you investigate.
    • 
          
  • Readiness probe pointing at a changed endpoint path that no longer exists in the new version.
    • 
          
  • Not enough cluster capacity for maxSurge extra Pods during the rollout.
    • 
        

      

    

  


  
  
  
  

    

      ☸️ kubernetes · 09 · ❓ interview prep
      
Interview Q&A — Kubernetes

      
The questions that separate "used kubectl a bit" from "actually understands how this works."

    


    

      
Q.Deployment vs StatefulSet vs DaemonSet — when would you use each?▾

      

        
A: A Deployment manages identical, interchangeable Pods for stateless apps — any Pod can be replaced by any other, and Pod names/storage aren't stable across restarts. A StatefulSet gives each Pod a stable, predictable identity (name, network identity, and its own PersistentVolume) — essential for databases and clustered apps that care about identity (e.g. postgres-0, postgres-1). A DaemonSet ensures exactly one Pod runs on every (or selected) node — used for node-level agents like log collectors or monitoring exporters.

      

    


    

      
Q.How does a Service know which Pods to send traffic to?▾

      

        
A: A Service has a selector (a set of label key/value pairs). Kubernetes continuously watches for Pods whose labels match that selector and maintains an Endpoints (or EndpointSlice) object listing their IPs. kube-proxy on every node uses this list to set up routing rules (iptables/IPVS) so traffic sent to the Service's ClusterIP gets load-balanced across those Pod IPs. A Pod only gets added to the Endpoints list if its readiness probe is passing.

      

    


    

      
Q.Explain the difference between RollingUpdate and Recreate deployment strategies.▾

      

        
A: RollingUpdate (default) gradually replaces old Pods with new ones, controlled by maxUnavailable (how many can be down at once) and maxSurge (how many extra Pods can be created temporarily) — achieves zero-downtime deployments. Recreate terminates ALL existing Pods first, then creates new ones — causes downtime but is necessary when the old and new versions can't safely run simultaneously (e.g. incompatible database schema migrations, or apps that can't share a singleton resource).

      

    


    

      
Q.What happens if you run kubectl delete pod on a Pod managed by a Deployment?▾

      

        
A: The Pod is deleted, but the Deployment's ReplicaSet controller notices the actual replica count (now N-1) no longer matches the desired count (N), and immediately creates a brand new Pod to replace it. The new Pod gets a different name/IP. This is the "self-healing" reconciliation loop in action — to truly remove a Pod permanently, you'd scale down the Deployment itself.

      

    


    

      
Q.What's the difference between a ConfigMap and a Secret?▾

      

        
A: Functionally they're very similar — both store key/value data consumable as env vars or mounted files. The difference is intent and handling: Secrets are meant for sensitive data (passwords, tokens, certs), are base64-encoded (not encrypted by default — encryption-at-rest must be configured separately), and Kubernetes treats them with extra care (e.g. not shown in plaintext via kubectl describe, can be restricted via RBAC more tightly). ConfigMaps are for non-sensitive configuration.

      

    


    

      
Q.How does the Horizontal Pod Autoscaler work under the hood?▾

      

        
A: The HPA controller periodically (every ~15s by default) queries the metrics-server for current resource usage (e.g. average CPU utilization across Pods of a Deployment). It compares this to the target you set (e.g. 70% CPU) and calculates a new desired replica count proportionally, then updates the target Deployment/ReplicaSet's replicas field — which then triggers normal scheduling of new Pods. It respects minReplicas/maxReplicas bounds and has built-in stabilization windows to avoid rapid flapping.

      

    


    

      
Q.What is etcd and why is it so critical?▾

      

        
A: etcd is a distributed, consistent key-value store that holds the entire state of the cluster — every object definition (Pods, Deployments, Secrets, ConfigMaps, etc.) lives there. The API server is the only component that talks to etcd directly. If etcd is lost without backups, the cluster effectively loses all its configuration and state — which is why regular etcd backups are a critical production responsibility.

      

    


    

      
Q.Liveness vs Readiness probe — what's the practical difference?▾

      

        
A: A failing liveness probe causes kubelet to restart the container — use it to detect "stuck/deadlocked" states that only a restart fixes. A failing readiness probe just removes the Pod from the Service's Endpoints (so it stops receiving traffic) without restarting it — use it for "temporarily can't serve traffic" states like still warming up a cache or briefly losing a DB connection. Mixing these up is a classic mistake: using a DB-dependency check as a liveness probe can cause a restart storm cluster-wide if the DB has a brief outage.

      

    


    

      
Q.Walk through how you'd debug a Pod that's not starting.▾

      

        
A: First kubectl get pods to see the status (Pending, ImagePullBackOff, CrashLoopBackOff, etc — each points to a different category of issue). Then kubectl describe pod <pod> — the Events section at the bottom almost always tells you exactly what's wrong (scheduling failure, image pull error, probe failures, volume mount issues). If the container did start but crashed, kubectl logs <pod> --previous shows the crashed instance's logs. For deeper issues, exec into a similar debug Pod on the same node/network to test connectivity manually.

      

    


    

      
Q.What's the difference between namespaced and cluster-scoped resources?▾

      

        
A: Namespaced resources (Pods, Deployments, Services, ConfigMaps, Secrets, etc.) exist within a namespace — their names only need to be unique within that namespace, and RBAC Roles can scope access to just one namespace. Cluster-scoped resources (Nodes, PersistentVolumes, Namespaces themselves, ClusterRoles, StorageClasses) exist outside any namespace and are visible/relevant cluster-wide. You can check with kubectl api-resources --namespaced=true (or false).

      

    


    

      
Q.How does Pod-to-Pod networking work across different nodes?▾

      

        
A: Kubernetes requires a flat network model where every Pod gets its own unique IP, and any Pod can reach any other Pod's IP directly without NAT — regardless of which node they're on. This is implemented by a CNI (Container Network Interface) plugin (e.g. Calico, Flannel, Cilium), which sets up routing/overlay networks between nodes to fulfill this model. Services and kube-proxy then sit on top of this flat Pod network to provide stable virtual IPs and load balancing.
